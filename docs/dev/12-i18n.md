# 12 — Internationalization (i18n)

ELMS supports three languages out of the box: English (`en`), Arabic (`ar`), and French (`fr`). Arabic uses right-to-left (RTL) layout. This document covers the i18next setup, namespace structure, adding new translation keys, adding a new language, RTL support, font loading, and the audit tool.

## Supported Languages

| Code | Language | Direction | Default? |
|------|----------|-----------|---------|
| `ar` | Arabic | RTL | Yes (default locale) |
| `en` | English | LTR | Fallback locale |
| `fr` | French | LTR | — |

The default locale on first load is `ar`. This can be overridden at build time by setting the `VITE_APP_LOCALE` environment variable.

---

## Library and Configuration

The frontend uses [i18next](https://www.i18next.com/) with the [react-i18next](https://react.i18next.com/) binding.

The i18n instance is initialized in `packages/frontend/src/i18n/index.ts` and imported side-effectfully in `main.tsx`:

```typescript
import "./i18n";
```

The initialization call:

```typescript
void i18n.use(initReactI18next).init({
  lng: getInitialLanguage(),
  fallbackLng: "en",
  interpolation: { escapeValue: false },
  resources: {
    ar: { auth: authAr, app: appAr },
    en: { auth: authEn, app: appEn },
    fr: { auth: authFr, app: appFr }
  }
});
```

All locale resources are imported statically — there is no dynamic loading at runtime. Translation JSON files are bundled at build time and separated into per-locale chunks by Vite (see [Vite Build Chunk Separation](#vite-build-chunk-separation)).

`fallbackLng: "en"` ensures that any key missing in the active locale falls back to English rather than showing the raw key string.

## Namespace Structure

Translations are split into two namespaces per language, each corresponding to a JSON file:

| Namespace | File | Purpose |
|-----------|------|---------|
| `auth` | `locales/<lang>/auth.json` | Login, register, setup, invite screens |
| `app` | `locales/<lang>/app.json` | All in-app UI strings (navigation, modules, tables, forms, errors) |

**Directory layout:**

```
packages/frontend/src/i18n/
├── index.ts
└── locales/
    ├── ar/
    │   ├── auth.json
    │   └── app.json
    ├── en/
    │   ├── auth.json
    │   └── app.json
    └── fr/
        ├── auth.json
        └── app.json
```

Keys within each namespace are nested objects. For example:

```json
// locales/en/app.json (excerpt)
{
  "cases": {
    "title": "Cases",
    "create": "New Case",
    "status": {
      "open": "Open",
      "closed": "Closed"
    }
  }
}
```

Reference the key in code as `"app:cases.status.open"`.

## Language Detection and Persistence

Language selection is persisted to `localStorage` under the key `elms-lang`. The `getInitialLanguage()` function in `i18n/index.ts` reads this key on startup:

```typescript
function getInitialLanguage(): string {
  try {
    const stored = localStorage.getItem("elms-lang");
    if (stored && ["ar", "en", "fr"].includes(stored)) return stored;
  } catch { /* localStorage may be unavailable */ }
  return (import.meta.env.VITE_APP_LOCALE as string | undefined) ?? "ar";
}
```

When the language changes at runtime, i18n fires a `languageChanged` event and the handler persists the new value:

```typescript
i18n.on("languageChanged", (lng) => {
  localStorage.setItem("elms-lang", lng);
});
```

## RTL Support

Arabic is a right-to-left language. RTL layout is handled by applying the `dir="rtl"` attribute to the document root when `ar` is active, managed by the `DirectionProvider` component in `packages/frontend/src/components/shared/DirectionProvider.tsx`.

CSS utilities for RTL are provided by [tailwindcss-rtl](https://github.com/20lives/tailwindcss-rtl), which adds logical property variants to Tailwind classes (e.g., `ms-4` for margin-inline-start instead of `ml-4`).

When writing styles that differ between LTR and RTL:
- Prefer **logical properties** (`ms-`, `me-`, `ps-`, `pe-`, `rounded-s-`, `rounded-e-`) over physical ones (`ml-`, `mr-`, `pl-`, `pr-`).
- Use `rtl:` and `ltr:` Tailwind variants for cases where logical properties are insufficient.

## Font Loading

Arabic requires a dedicated typeface for legibility. Two Arabic font packages are included:

| Package | Font | Usage |
|---------|------|-------|
| `@fontsource/cairo` | Cairo | Primary Arabic UI font |
| `@fontsource/noto-naskh-arabic` | Noto Naskh Arabic | Fallback / document rendering |

Fonts are loaded via a custom `fonts.css` file imported in `main.tsx` (`import "./fonts.css"`). The CSS uses `@font-face` declarations with self-hosted woff files. The comment in `main.tsx` notes: _"Custom @font-face — self-hosted woff files with woff-first for WebKitGTK"_ — this ensures compatibility with the WebKitGTK renderer used in the Linux Tauri build.

Do not use the default `@fontsource` CSS imports directly (e.g., `import "@fontsource/cairo/400.css"`), as those use woff2-first ordering which does not render correctly in WebKitGTK.

## Vite Build Chunk Separation

The Vite build is configured to emit Arabic and French locale resources as separate chunks. This means users who never switch away from English do not download Arabic or French translation JSON.

When adding a new locale, add it to the Vite manual chunks configuration in `packages/frontend/vite.config.ts` (or equivalent):

```typescript
manualChunks: {
  "locale-ar": ["./src/i18n/locales/ar/auth.json", "./src/i18n/locales/ar/app.json"],
  "locale-fr": ["./src/i18n/locales/fr/auth.json", "./src/i18n/locales/fr/app.json"]
}
```

English locale files are included in the main bundle since English is the fallback locale and always required.

## How to Add a New Translation Key

1. **Add the key to the English locale file first.** English (`en`) is the reference language.

   ```json
   // packages/frontend/src/i18n/locales/en/app.json
   {
     "notes": {
       "title": "Notes",
       "empty": "No notes yet.",
       "createFirst": "Create your first note to get started."
     }
   }
   ```

2. **Add the same key to `ar`:**

   ```json
   // packages/frontend/src/i18n/locales/ar/app.json
   {
     "notes": {
       "title": "الملاحظات",
       "empty": "لا توجد ملاحظات بعد.",
       "createFirst": "أنشئ ملاحظتك الأولى للبدء."
     }
   }
   ```

3. **Add the same key to `fr`:**

   ```json
   // packages/frontend/src/i18n/locales/fr/app.json
   {
     "notes": {
       "title": "Notes",
       "empty": "Aucune note pour l'instant.",
       "createFirst": "Créez votre première note pour commencer."
     }
   }
   ```

4. **Run the audit script** to confirm no keys are missing:

   ```bash
   pnpm tsx scripts/i18n-audit.ts
   ```

5. **Use the key in your component** (see [Using Translations in Components](#using-translations-in-components)).

## How to Add a New Language

1. **Create locale files** for the new language under `locales/<code>/`:

   ```
   packages/frontend/src/i18n/locales/de/
   ├── auth.json
   └── app.json
   ```

   Copy the English files as a starting point and translate all values.

2. **Import and register** the new locale in `i18n/index.ts`:

   ```typescript
   import authDe from "./locales/de/auth.json";
   import appDe from "./locales/de/app.json";

   void i18n.use(initReactI18next).init({
     // ...
     resources: {
       // ...existing locales...
       de: { auth: authDe, app: appDe }
     }
   });
   ```

3. **Update the language guard** in `getInitialLanguage()`:

   ```typescript
   if (stored && ["ar", "en", "fr", "de"].includes(stored)) return stored;
   ```

4. **Add to the i18n-audit script** — open `scripts/i18n-audit.ts` and add the new code to the `LANGS` array:

   ```typescript
   const LANGS = ["ar", "de", "en", "fr"] as const;
   ```

5. **Add a Vite manual chunk** for the new locale files (see [Vite Build Chunk Separation](#vite-build-chunk-separation)).

6. **Handle RTL** if the language is right-to-left: update `DirectionProvider` to apply `dir="rtl"` for the new locale code.

7. **Add a font** if the new language requires a non-Latin script.

## Using Translations in Components

Use the `useTranslation` hook from `react-i18next`. Specify the namespace as the first argument:

```typescript
import { useTranslation } from "react-i18next";

export function NotesPage() {
  const { t } = useTranslation("app");

  return (
    <div>
      <h1>{t("notes.title")}</h1>
      <p>{t("notes.empty")}</p>
    </div>
  );
}
```

For auth screens, use the `"auth"` namespace:

```typescript
const { t } = useTranslation("auth");
// t("login.emailLabel"), t("login.passwordLabel"), etc.
```

**Interpolation:**

```typescript
// Translation: "Welcome, {{name}}"
t("dashboard.welcome", { name: user.fullName })
```

**Plural forms:**

```typescript
// Translation: "{{count}} case" / "{{count}} cases"
t("cases.count", { count: 5 })
```

## i18n Audit Script

The `scripts/i18n-audit.ts` script compares all locale files against the English reference and reports missing keys.

```bash
# Report only (exit 0 even if keys are missing)
pnpm tsx scripts/i18n-audit.ts

# Fail with exit code 1 if any keys are missing (for CI)
pnpm tsx scripts/i18n-audit.ts --fail
```

Sample output when everything is complete:

```
auth.json — complete
app.json — complete

──────────────────────────────
All translations complete.
```

Sample output with missing keys:

```
app.json
  [ar] missing 2 key(s):
       • notes.title
       • notes.empty

──────────────────────────────
Total missing keys: 2
```

The script only checks keys present in `en` but absent in `ar` or `fr`. It does not flag extra keys in non-English locales.

For the full script documentation see [Scripts Reference — i18n-audit.ts](./10-scripts.md#i18n-auditts).
